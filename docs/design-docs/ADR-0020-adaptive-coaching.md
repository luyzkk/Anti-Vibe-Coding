# ADR-0020: Adaptive Coaching Framework (v6.3.0)

## Context

The Anti-Vibe Coding plugin treats all projects generically. Skills ignore the architecture
profile that the detector has produced since v5.3. On each invocation, skills re-discover
the project via local heuristics. There is no agent capability inventory for the project to
be audited against.

Three problems compound:
1. Skills give identical advice to a Next.js App Router project and a flat MVC project — the
   profile information exists but is never consumed.
2. No capability inventory means `/parity-audit` cannot exist — there is nothing to diff the
   tool registry against.
3. Stale-detection is absent — if a project's routes change, skills keep operating against an
   outdated mental model.

This ADR records the architectural decisions forming the Adaptive Coaching Framework (v6.3.0),
organized around three artifacts: `PrefaceContext` helper, `capabilities.json`, and
`parity-gaps.json`.

**PRD:** [Adaptive Coaching v6.3.0](../exec-plans/completed/2026-05-14-v6.3.0-adaptive-coaching/PRD.md)

## Decision

Ten decisions (D1–D10) define the framework.

### D1: Profile-aware preface via explicit block in SKILL.md

Skills that consume the adaptive profile declare an explicit block called
`## Profile-Aware Preface` in their `SKILL.md`. The block specifies which lookup table to use
and which fallback to apply when `readPrefaceContext` returns `profile: null`.

### D2: PrefaceContext shape `{ profile, language, framework, confidence }`

The shape is stable and extensible. `language` and `framework` are `null` in v6.3.0 and will
be populated in v6.5/v6.6 without breaking existing callers. Skills must NOT access
`archProfile.signals` directly — only the `PrefaceContext` shape.

### D3: Coverage limited to nextjs-app-router (AST) and mvc-flat (LLM-fallback) in v6.3.0

The AST parser covers only `app/**/route.ts` (nextjs-app-router). The mvc-flat profile uses
LLM-fallback, marked with `source: "llm"`. The three remaining profiles
(clean-architecture-ritual, vertical-slice, unknown-mixed) have no capability generation in
v6.3.0. Scope is controlled for the first cycle; v6.4+ extends coverage.

### D4: AST-first + LLM-fallback marked with source and confidence

Every entry in `capabilities.json` declares `source: "ast" | "llm"`. Entries with
`source: "llm"` have `confidence < 1.0` by definition. Auditors and skills must prioritize
`ast` entries and flag `llm` entries as "requires human review."

### D5: `/parity-audit` as a new independent skill (not an extension of `qa-visual`)

`/parity-audit` is an independent skill with `kind: "audit"`. `qa-visual` is refactored to
consume the `tool-registry-inspector` lib without UX changes (Plano 03, fase-03). The two
skills have distinct contracts: `qa-visual` focuses on visual regression, `parity-audit`
focuses on capability gaps.

### D6: Build-time default + stale checksum + silent warning

`capabilities.json` is generated by `/init` (build-time). The stale detector computes a
checksum of key paths (`package.json`, top-level `src/`) and emits a silent (non-blocking)
warning when the checksum diverges from the snapshot stored in `capabilities.json`.

### D7: `discovery/` folder for capabilities, parity-gaps, and schemas

All discovery artifacts live in `discovery/` at the project root. The subfolder
`discovery/_schemas/` holds the versioned JSON Schemas. Runtime JSONs live directly in
`discovery/`.

### D8: `discovery/*.json` gitignored by default

Runtime artifacts (`capabilities.json`, `parity-gaps.json`) are gitignored — they are
generated locally and may diverge between machines. Schemas (`discovery/_schemas/`) are
committed.

### D9: `qa-visual` migrates to `tool-registry-inspector` without UX changes

The `tool-registry-inspector` lib enumerates MCPs, builtin tools, and subagents. `qa-visual`
uses this lib internally. Users perceive no difference — external behavior is identical to v6.2.

### D10: Stale checksum covers key paths (package.json, top-level src/)

The stale detector checks: `package.json` (dependencies), every file/folder directly under
`src/` (top-level only, non-recursive), and `routes/` if present. This strategy is
conservative: it prefers false-positives (warning when unnecessary) over false-negatives
(no warning when capabilities are stale).

## Alternatives Considered

**D1 — inject profile via environment variable at runtime.**
Rejected: creates invisible coupling; makes unit testing hard because tests must set env vars
to exercise profile branches.

**D2 — expose `ArchitectureProfile` directly to skills.**
Rejected: exposes internal detector fields (`signals`, `detectedAt`) that skills do not need
and that may change without notice.

**D5 — extend `qa-visual` with a `--parity` mode.**
Rejected: mixes responsibilities and breaks the single-skill-single-responsibility principle.
`qa-visual` is about visual regression; `parity-audit` is about capability gaps.

**D6 — regenerate `capabilities.json` on every skill invocation.**
Rejected: latency cost is unjustifiable for the majority of invocations where the project has
not changed. Build-time generation with stale detection covers the failure mode at near-zero
steady-state cost.

## Consequences

Positive:
- Skills become profile-aware without duplicating profile-reading logic.
- Capability inventory enables automated parity audit (gap analysis vs tool registry).
- Stable `PrefaceContext` shape allows gradual evolution (v6.5/v6.6) without breaking callers.
- `discovery/` folder provides a clear, auditable location for all discovery artifacts.

Negative / Trade-offs:
- Partial coverage in v6.3.0: only 2 of 5 profiles have `capabilities.json` generation.
- `discovery/*.json` gitignored creates local state — CI must generate the file before consuming it.
- Stale detection via checksum incurs IO cost on every skill invocation (mitigated: key paths only, not full tree).
- LLM-sourced entries in `capabilities.json` require human review; automated consumers must
  handle the lower-confidence entries explicitly.

## Reversibility

Largely reversible:
- `skills/lib/preface-context.ts` is additive — removing it requires only deleting the file
  and removing calls from skills that adopt it. No data is lost.
- `discovery/_schemas/` are metadata only — deleting them has no runtime impact.
- `discovery/*.json` are generated artifacts — they can be regenerated or deleted without
  data loss (they carry no source of truth).
- The `## Profile-Aware Preface` block in each SKILL.md is documentation — removing it
  degrades adaptive behavior but does not break the skill.
- Irreversible aspect: if skills written for v6.4+ depend on `PrefaceContext`, removing the
  helper breaks those skills. Cost of reversal scales with adoption.
