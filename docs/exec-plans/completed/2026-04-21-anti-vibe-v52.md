# Anti-Vibe Coding v5.2 — Principios Akita

## Goal

O Anti-Vibe Coding v5.1 cobria bem o ciclo de construcao de software mas tinha tres lacunas criticas identificadas ao confrontar o plugin com evidencia empirica de 692 commits e 4 projetos reais em producao:

1. **O pipeline terminava onde o software comecava** — `verify-work` era o fim do fluxo. 125 dos 274 commits de um projeto real aconteceram *apos* o deploy. O plugin nao cobria a vida pos-producao.

2. **Regras de codigo limpo para agentes estavam ausentes ou conflitantes** — a politica "sem comentarios por padrao" contradiz diretamente o que Akita mostra: comentarios de proveniencia sao contexto de primeira classe para o agente. Sem limites de tamanho de arquivo/funcao, sem auditoria de nomes grepaveis, sem thresholds de coverage definidos.

3. **A dinamica humano-agente nao estava ensinada** — o principio "humano navega, agente pilota" nao existia como skill ensinavel.

A v5.2 adiciona a **quarta pata** do mantra: _"Ensinar antes de codar. Planejar antes de executar. Verificar antes de entregar. **Iterar em producao.**"_

## Scope

**Entregues (Must Have + Should Have):**
- 2 hooks novos: `file-size-guard` (PostToolUse — avisa arquivo >500 linhas ou funcao >40 linhas) e `grepping-names` (PreToolUse git commit — avisa nomes genericos com >10 hits)
- Politica de comentarios D3 (WHY sim, WHAT nao, nao podar comentarios do agente) em `rules/code-quality.md` e `CLAUDE.md`
- Thresholds TDD hardcoded: >=95% biz logic, >=80% global, >=70% branch coverage em `rules/testing-standards.md`
- 4 skills standalone pos-deploy: `/incident-response`, `/defensive-patterns`, `/centralize-config`, `/pair-programming-with-agent`
- Skill `/iterate` — ciclo pos-deploy: incident response + regression fix + hardening
- `/verify-work` integrado com sugestoes de regression-fix e hardening
- `/init` com template Akita (5 secoes: Code Style, Comments, Tests, Dependencies, Logging) multi-linguagem TS/JS + Python + Ruby
- `/update` gerenciado: diff por secao + confirmacao antes de aplicar (CA-07)
- `anti-vibe-review`: novo item "Nomes grepaveis" no checklist
- `hooks/hooks.json`: tabela Akita "Faz BEM / Faz MAL" no advisor hook SessionStart
- `rules/code-quality.md` e `rules/testing-standards.md` com exemplos Python + Ruby

**Won't Have desta versao:**
- Modo conversational execute (v6.0)
- Go/Rust nos exemplos multi-linguagem
- Mutation testing no verify-work
- Auto-update via npx/bun

## Assumptions

- Plugin Anti-Vibe Coding v5.1.0 ja instalado e `hooks.json` disponivel
- `anti-vibe-coding/` e repositorio git independente (GT-03) — commits com CWD dentro desse diretorio
- Evidencia empirica de Fabio Akita (692 commits, 4 projetos reais, 274 commits pos-deploy) valida as decisoes de threshold e politica de comentarios
- Hooks `.cjs` — usar `require()` e `module.exports` (nao `import`); `process.exit(2)` proibido para warnings
- autocrlf ativo no Windows — LF→CRLF em todos os commits; normal, nao afeta execucao Node

## Compound Opportunity

> _(reconstructed: pre-v6 plan had no Compound Engineering section)_

O padrao de hook `.cjs` estabelecido no Plano 01 (stdin JSON → stdout warning → exit 0) tornou-se o padrao canônico para todos os hooks subsequentes do plugin. Os arquivos `file-size-guard.cjs` e `grepping-names.cjs` serviram de referencia para hooks criados em v5.3 e v6.0.

A politica D3 (WHY sim, WHAT nao) foi o primeiro principio Akita formalizado no plugin e tornou-se fundacao para os 5 principios universais integrados na v5.3 (Plano 04 da v5.3 reusa a politica de comentarios como Principio #5 Comment Provenance).

## Lessons Captured

### Do plano01/MEMORY.md

- **GT-01:** autocrlf ativo no Windows — LF→CRLF em todos os commits. Normal, nao afeta execucao Node.js. Nao confundir com mudancas reais.
- **GT-02:** `grep -c` retorna exit code 1 quando count=0 (nao encontrou pattern). Verificacao de `process.exit(2)` via contagem funciona corretamente.
- **GT-03:** `anti-vibe-coding/` e repositorio git independente — commits com CWD dentro desse diretorio.
- Padrao de hook `.cjs` estabelecido: `process.stdin.on('end', run)` para PreToolUse / `process.nextTick(run)` para PostToolUse.

### Do plano02/MEMORY.md

- Politica D3 (WHY/WHAT) agora esta em 3 lugares: `rules/code-quality.md`, `rules/testing-standards.md` (coverage thresholds D7), e `anti-vibe-coding/CLAUDE.md`. Planos futuros devem REFERENCIAR essas regras, nao duplica-las.
- Threshold de funcao eh agora 40L (nao 100L). Skills de auditoria devem usar 40L como referencia.

### Do plano03/MEMORY.md

- 4 skills criadas — nenhuma registrada em `plugin-manifest.json` ainda ate o Plano 04 fase-03 registrar todas de uma vez.
- Skills devem ter <=200 linhas (maximo observado: 190L em pair-programming-with-agent).
- `centralize-config` cross-referencia `defensive-patterns` categoria 7 — link bidirecional intencional.

### Do plano04/MEMORY.md

- **DI-01 (fase-01):** iterate/SKILL.md ficou 215L vs estimativa de 200L. File-size-guard nao alertou (threshold eh 500L para arquivos, 40L para funcoes). Estimativas de linhas com exemplos multi-linguagem subestimam.

### Do plano05/MEMORY.md

- **GT-02 (fase-01):** Arquivo final init/SKILL.md ficou 625L (spec previa ~580L). Secoes multi-linguagem mais longas que estimado. Aceitavel.
- Padrao backtick aninhado em SKILL.md: usar quadruple backticks (`````markdown`) como fence externo quando a secao contem triple backticks internos.
- CA-07 documentado formalmente em `skills/update/SKILL.md` Regras Importantes item 8.

### Do plano06/MEMORY.md

- **DI-01 (fase-02):** Caractere `→` (U+2192) mantido no JSON do hooks.json — JSON parser suporta Unicode sem problema.
- **DI-02 (fase-03):** Quadruple backtick necessario APENAS em SKILL.md com blocos de codigo aninhados, nao em rules/*.md diretos.
